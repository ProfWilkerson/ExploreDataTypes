Explore: What Type of Data Is This?

Quick Classification Practice

CLASSIFY & COMPARE  |  NO SUBMISSION REQUIRED

Purpose: Before AI systems can learn from data, that data has to be understood and prepared. One of the first steps is recognizing what type of data you are working with, because structured, unstructured, and semi-structured data each require different handling. This exploration builds quick, confident data-type recognition.

What to do: Read each of the 12 data examples below. For each one, click the button that best matches your classification. When you have worked through all of them, reveal the answer key at the bottom to compare your thinking.

QUICK REFERENCE

Data Type What It Looks Like Key Signal
Structured Organized into rows and columns with a fixed schema; lives comfortably in a spreadsheet or relational database Every record has the same fields in the same order
Unstructured Free-form content with no predefined model; includes text documents, images, audio, and video No consistent fields or tags a computer can directly query
Semi-Structured Has some organizational markers (tags, keys, headers) but does not conform to a rigid table schema Self-describing format like JSON, XML, or HTML

CLASSIFY EACH EXAMPLE

Example 1

A hospital database table with columns for Patient ID, Visit Date, Blood Pressure, Heart Rate, and Temperature. Every patient visit adds one new row, and the columns never change.

Example 2

A folder containing 400 MP3 recordings from a customer service call center. Each file is a phone conversation between a representative and a customer.

Example 3

A JSON file that logs each online order. Every order has fields for order_id, customer_email, and timestamp, plus an "items" field that holds a variable-length list of products and quantities.

Example 4

A stack of paper intake forms that patients completed by hand in a waiting room. The forms have been scanned and saved as image files (PNG).

Example 5

A spreadsheet tracking weekly employee attendance across three departments. Each row is one employee; columns represent each day of the work week, with values of "Present," "Absent," or "Remote."

Example 6

A collection of emails sent to a retailer's customer support inbox. Each message is written differently — varying length, tone, and topic — with no consistent fields or format.

Example 7

An XML product catalog where every item is wrapped in consistent tags: <name>, <price>, <category>, and <stock_count>. Some items also include an optional <description> tag; others do not.

Example 8

Handwritten clinical notes taken by a physician during a patient consultation. The notes include observations, impressions, and a treatment plan written in the physician's own words and shorthand.

Example 9

A CSV file exported from a point-of-sale system. The file has a header row with column names (Date, Store, Item, Quantity, Unit Price, Total) followed by one row per transaction.

Example 10

A dataset of social media posts. Each post contains free-form text written by a user, alongside system-generated metadata fields: username, timestamp, post ID, hashtags, and location tag.

Example 11

A set of company web pages for a product catalog. Each page uses HTML tags to define a product title, a price, and a navigation menu, but the product descriptions are written as free-form paragraphs of varying length.

Example 12

A relational database table in a retail inventory system. Columns include SKU, Product Name, Category, Unit Price, and Quantity on Hand. Every product record follows the exact same schema.

ANSWER KEY
1 Structured Fixed columns, consistent rows, and a defined schema make this a textbook relational database table.
2 Unstructured Audio files contain no rows, columns, or tags. The spoken content cannot be directly queried without additional processing such as speech-to-text transcription.
3 Semi-Structured JSON uses key-value pairs to provide organization, but the variable-length "items" list means records do not all share the same flat structure required for a relational table.
4 Unstructured Scanning a paper form creates an image. The handwriting within it is not machine-readable; a computer sees pixels, not fields or values, until OCR or similar processing is applied.
5 Structured Every employee record occupies one row, and every column represents the same field across all rows. This uniform schema is the hallmark of structured data.
6 Unstructured Email body text is free-form prose. While email headers (To, From, Subject) have some structure, the actual message content has no consistent schema a system can parse or query directly.
7 Semi-Structured XML provides self-describing tags that organize the data, but the optional description tag means not every record is identical. This flexibility is characteristic of semi-structured formats.
8 Unstructured Handwritten narrative notes have no consistent structure. Each note reflects the individual clinician's style and contains free-form text that cannot be directly parsed without NLP.
9 Structured A CSV with a consistent header row and uniform columns is structured data. Every transaction record contains the same fields, making it easy to load into a database or analysis tool.
10 Semi-Structured The metadata fields (username, timestamp, hashtags) are consistent and machine-readable, but the post text itself is free-form. This mix of structured metadata and unstructured content is a defining feature of semi-structured data.
11 Semi-Structured HTML tags give the pages a defined skeleton (title, price, navigation), but the free-form description paragraphs mean the content does not conform to a fixed schema. HTML is a classic semi-structured format.
12 Structured Relational database tables are the standard example of structured data. Every product record follows the exact same schema, with no optional or variable fields.
